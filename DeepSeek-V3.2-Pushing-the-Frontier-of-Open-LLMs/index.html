DeepSeek-V3.2: Pushing the Frontier of Open Large Language Models

DeepSeek-AI · arXiv:2512.02556 · 2025/12 · 中文深度解读
一句话概括. DeepSeek-V3.2 在 V3.1-Terminus 基础上继续训练，引入 DeepSeek Sparse Attention (DSA) 把核心注意力复杂度从 $O(L^2)$ 降为 $O(Lk)$；并把 RL 后训练的 compute budget 拉到 预训练成本的 10%+，配合"specialist 蒸馏 + 混合 GRPO"和 1827 个合成 agent 环境 / 8.5 万复杂提示词 的 RL 数据管线，使开源模型在推理基准上接近 GPT-5、在 agent 基准上把开源/闭源差距大幅缩小。其 high-compute 变体 Speciale 在 IMO/IOI/CMO/ICPC WF 2025 上拿到金牌。

摘要 / 核心贡献

论文将开源大模型在复杂任务上落后于闭源前沿模型的原因归结为三点缺陷，并对每一点给出针对性方案：

  1. 架构层面：Vanilla / MLA Attention 在长序列上是 $O(L^2)$，制约部署与后训练。→ DSA 让核心注意力变成 $O(Lk)$，长上下文 decoding 成本下降数倍。
  2. 资源层面：开源模型后训练 compute 投入不足。→ Scalable GRPO 通过 unbiased KL、off-policy mask、Keep Routing、Keep Sampling Mask 把 RL 训得稳，跑出 >10% pre-train cost 的后训练算力。
  3. Agent 层面：开源模型在工具调用 / 复杂指令下泛化差。→ Synth Pipeline 自动合成 1827 个 environment、4417 个任务、8.5 万 prompt，把推理与工具调用统一在一条轨迹里。
Figure 1. 模型对比基准：左半边 Reasoning（AIME 2025、HMMT 2025、HLE、Codeforces Rating），右半边 Agentic（SWE-Verified、Terminal Bench 2.0、$\tau^2$ Bench、Tool Decathlon）。柱形从左到右分别是 V3.2-Speciale (实心蓝)、V3.2-Thinking (条纹蓝)、GPT-5-High、Claude-4.5-Sonnet、Gemini-3.0-Pro。HMMT 2025 用的是 February 比赛、HLE 是 text-only。Codeforces 用右轴的 Rating 数值。可以看出：在 Reasoning 上 Speciale 全面追平或超越 Gemini-3.0-Pro；Agentic 上 V3.2-Thinking 在 SWE Verified、Terminal Bench、Tool Decathlon 上超越 GPT-5-High 与 Claude-4.5-Sonnet（Terminal Bench 46.4 显著高于 GPT-5 的 35.2）。

1 背景与核心问题

2024-2025 年开源 LLM（Qwen3、Kimi-K2、MiniMax M2、GLM-4.5）持续进步，但闭源 GPT-5 / Gemini-3 / Claude-4.5 进步更快，差距在 拉大而非收敛。论文判断核心瓶颈不在"再多堆点参数 / token"，而在三件事：

  • 稠密 attention 的 $O(L^2)$ 让 128K 推理与 RL rollout 极其昂贵；
  • 开源团队普遍 RL 算力远不及 pre-train（典型 1-3%），高难任务训不动；
  • 非 SWE-Verified / 标准基准之外的"长尾" agent 任务（MCP-Universe、Tool Decathlon）开源模型表现远弱。

2 DeepSeek Sparse Attention (DSA)

2.1 原型：Lightning Indexer + Top-k Selector

DSA 的思路：在每个 query token 真正去做 attention 之前，先用一个非常便宜的 lightning indexer 给历史 token 打分，然后只在 top-k 个最相关的 KV 上做 attention。它由两件事组成。

(1) Lightning Indexer. 对 query $\mathbf h_t$ 与历史 token $\mathbf h_s$ 计算索引分数：

$$ I_{t,s} = \sum_{j=1}^{H^I} w_{t,j}^I \cdot \mathrm{ReLU}\!\left(\mathbf q_{t,j}^I \cdot \mathbf k_s^I\right). \quad (1) $$

每个项的含义：$\mathbf q_{t,j}^I, w_{t,j}^I$ 由 $\mathbf h_t$ 经 indexer 投影得到的第 $j$ 个 head 的 query 向量与门控权重；$\mathbf k_s^I$ 是 $\mathbf h_s$ 投影得到的 indexer key。用 ReLU 而不是 softmax 是为了 throughput——可以走 FP8 高吞吐 kernel；同时 $H^I$ 设很小，使 indexer 整体耗时远小于 MLA。

(2) Top-k Token Selection. 对每个 query 取 top-k 索引最高的 KV 做注意力：

$$ \mathbf u_t = \mathrm{Attn}\!\left(\mathbf h_t,\; \{\mathbf c_s \mid I_{t,s}\in\mathrm{Top\text{-}k}(I_{t,:})\}\right). \quad (2) $$

关键直觉：在 128K 上下文里真正 attention 高的 token 通常 极其稀疏，所以 $k\ll L$（论文用 $k=2048$）即可在性能不掉的前提下把 attention 复杂度从 $O(L^2)$ 砍到 $O(Lk)$。

2.2 在 MLA 上落地：MQA 模式

为了让 DSA 能从 V3.1-Terminus 的 MLA 检查点 continue training（不是从头训），并满足"每个 KV 必须被多个 query 共享"的 kernel 效率约束，作者把 DSA 实例化在 MLA 的 MQA 模式之上：每个 latent KV 被该 token 的全部 query head 共享。

Figure 2. DSA 在 MLA 上的实例化。下方 input hidden $\mathbf h_t$ 经过两条平行通路：左侧深色路径走 MLA 主干——$\mathbf c_t^Q$ 投影出 $\mathbf q_{t,i}^A$ 与 $\mathbf q_{t,i}^R$（apply RoPE 后拼接），KV 端 $\mathbf c_t^{KV}$ 与 $\mathbf k_t^R$ 拼接进入 Multi-Query Attention（Core Attention）；右侧绿色路径是新增的 Lightning Indexer——$\{\mathbf q_{t,j}^I\}$、$\mathbf k_t^I$、门控权重 $\{w_{t,j}^I\}$ 经 partial RoPE 与点积得到索引分数，送到中间的 Top-k Selector，由它在 KV cache 里筛出 $k$ 条 latent，再交给 core attention。绿色 = 新增；灰色 = 原 MLA 路径。这张图回答的是"DSA 是怎么挂在已有 MLA 上的，能否平滑 continue training"。

2.3 两阶段 continued pre-training

从 128K context 的 V3.1-Terminus base 出发：

Stage 1 — Dense Warm-up（仅训 indexer）. 冻结所有参数只训 indexer。把主 attention 的所有 head 求和后做 L1-norm 得到 target 分布 $p_{t,:}$，让 indexer 输出的 softmax 与之 KL 对齐：
$$ \mathcal L^I = \sum_t \mathbb D_{\mathrm{KL}}\!\left(p_{t,:}\,\|\,\mathrm{Softmax}(I_{t,:})\right). \quad (3) $$
LR=$10^{-3}$，1000 步 × 16×128K ≈ 2.1B tokens。
Stage 2 — Sparse Training（全参数训）. 引入 top-k 选择，主模型用 LM loss 优化全部参数，indexer 仍用 KL 对齐，但 只对被选中的 token 集合 $\mathcal S_t$ 求和：
$$ \mathcal L^I = \sum_t \mathbb D_{\mathrm{KL}}\!\left(p_{t,\mathcal S_t}\,\|\,\mathrm{Softmax}(I_{t,\mathcal S_t})\right). \quad (4) $$
关键工程细节：indexer 输入从计算图 detach，让 indexer 仅由 $\mathcal L^I$ 驱动，主模型仅由 LM loss 驱动；LR=$7.3\times 10^{-6}$；$k=2048$；15000 步 × 480×128K ≈ 943.7B tokens。

2.4 推理成本：H800 实测

Figure 3a (Prefilling). 每百万 token 的美元成本随 token 位置变化。蓝线 V3.1-Terminus 在 128K 处达到约 0.66 美元，几乎线性增长（$O(L^2)$ 分摊到每 token 后是 $O(L)$）；橙线 V3.2 在 128K 处仅约 0.18 美元，曲线接近平坦——这就是把核心 attention 从 $O(L^2)$ 降到 $O(Lk)$ 后剩下的 indexer + 其余开销。短序列（≤8K）时 V3.2 用 masked MHA 模拟 DSA 反而比稀疏路径更快，所以两线在原点附近交汇。
Figure 3b (Decoding). Decoding 阶段 V3.1-Terminus 的成本随 token 位置近似线性增长（每步 attention 都要扫整个 KV cache），128K 处接近 2.4 美元/M tokens；V3.2 因每步只 attend top-k = 2048 个 KV，曲线明显更平，128K 处 < 0.5 美元/M tokens——这是 DSA 在长上下文 agent 场景下最大的实际收益。

3 后训练：把 RL Compute 推到预训练的 10%+

后训练保持 V3.2-Exp 的两段式：先 specialist distillation（在 6 个领域：数学、编程、通用逻辑、通用 agent、agentic coding、agentic search 各训一个 specialist，再蒸馏回 base），再 mixed GRPO。但要把 RL compute 拉到 pre-train 的 10%+ 而不崩，需要 4 个稳定性技巧。

3.1 Scaling GRPO 的四个稳定性技巧

原始 GRPO 目标：

$$ \mathcal J_{\mathrm{GRPO}}(\theta) = \mathbb E\!\left[\frac{1}{G}\sum_i \frac{1}{|o_i|}\sum_t \min\!\left(r_{i,t}\hat A_{i,t},\,\mathrm{clip}(r_{i,t},1\!-\!\varepsilon,1\!+\!\varepsilon)\hat A_{i,t}\right) - \beta\,\mathbb D_{\mathrm{KL}}(\pi_\theta\|\pi_{\mathrm{ref}})\right]. \quad (5) $$

$\hat A_{i,t}=R_i-\mathrm{mean}(\boldsymbol R)$ 是组内归一化优势。下面四条技巧都是为了把 long-horizon、long-rollout、MoE 路由变化等带来的不稳定性压住。

(i) Unbiased KL Estimate. 把 Schulman 的 K3 估计器做 importance-sampling 修正：
$$ \mathbb D_{\mathrm{KL}}(\pi_\theta\|\pi_{\mathrm{ref}}) = \frac{\pi_\theta}{\pi_{\mathrm{old}}}\!\left(\frac{\pi_{\mathrm{ref}}}{\pi_\theta} - \log\frac{\pi_{\mathrm{ref}}}{\pi_\theta} - 1\right). \quad (7) $$
Why 关键：原 K3 估计器在 $\pi_\theta\ll\pi_{\mathrm{ref}}$（探索到罕见 token）时会赋予 unbounded 大权重，让梯度 explode。修正后梯度 unbiased。同时论文发现不同领域适合不同 KL 强度，数学任务甚至完全去掉 KL 反而更好。
(ii) Off-Policy Sequence Masking. 大 batch rollout + 多 mini-batch 更新天然 off-policy；inference / training 的 kernel 不一致还会再放大。引入 binary mask $M_{i,t}$：
$$ M_{i,t}=\begin{cases}0 & \hat A_{i,t}<0\;\text{且 sequence-level KL} > \delta \\ 1 & \text{otherwise}\end{cases}. \quad (9) $$
只对负优势 + 强 off-policy 的序列丢弃。直觉：模型最该从"自己刚犯过的错"中学，过度 off-policy 的负样本反而误导。
(iii) Keep Routing. MoE 模型 inference / training 框架的浮点差会让相同输入选不同 expert，激活子空间突变 → 训练不稳。训练时强制复用 inference 阶段记录下来的 routing 路径，让被优化的就是 sampling 时被激活的那批 expert 参数。这条从 V3-0324 起就在用。
(iv) Keep Sampling Mask. Top-p / top-k 截断改变了 $\pi_{\mathrm{old}}$ 与 $\pi_\theta$ 的 action space，违反 IS 前提。把 sampling 时的截断 mask 也带到 training，保证两个 policy 同 action subspace。Top-p + 该策略对 RL 训练后语言一致性 的保留尤其关键。

3.2 工具调用中的 Thinking：上下文管理

R1 的策略是"每出现新 message 就丢弃 reasoning"。在 tool-use 场景下这非常浪费——每次 tool 返回都要重推一遍整个推理链。V3.2 改成两条规则：

  1. 只有遇到 新的 user message 才丢弃历史 reasoning；如果只是 tool output，reasoning 全部保留。
  2. 即便丢弃了 reasoning trace，tool calls 与 tool results 的历史仍保留。

但 Roo Code / Terminus 这类把 tool 模拟成 user message 的框架不能从这条规则中受益（每轮都丢 reasoning），所以官方建议在它们之上跑 non-thinking 模式。

Figure 4. Thinking retention 机制示意。蓝色虚线框内是同一个 user message（User message 1）的多轮 tool 交互。Turn 1.1 输入 Tools+User message 1，输出 Thinking 1.1+Tool call 1.1；Turn 1.2 把 Thinking 1.1+Tool call 1.1+Tool result 1.1 全部带进 input 继续推理输出 Thinking 1.2+Tool call 1.2；Turn 1.3 同理累积全部 thinking/tool history 直到给出 Answer 1。蓝色 Thinking 块在多轮 tool 交互里持续保留。直到 Turn 2.1 出现 User message 2（蓝框右侧虚线箭头）：input 中所有 Thinking 1.x 被清空，但 Tool call 1.1/1.2 + Tool result 1.1/1.2 + Answer 1 仍保留。这张图直观回答"哪些 token 在哪一刻被丢弃"。

四类任务的关键设计差异：

  • Code agent. 从 GitHub mine 出 issue ↔ PR pair；用 LLM-judge 过滤；自动 setup environment，要求 F2P>0 且 P2F=0（gold patch 修了 issue 且没引入回归）才算成功。覆盖 Python/Java/JS/TS/C/C++/Go/PHP，几万个可重现环境。

  • Code interpreter. 数学 / 逻辑 / 数据科学题，必须借助 Jupyter 执行才能解。

4.4 Search Agent 的上下文管理

BrowseComp 类任务即便 128K 也常在推理途中爆窗。论文比较四种 test-time compute 扩展策略（占比超过 80% 时触发）：

  • Summary：把溢出轨迹总结后重新 rollout。
  • Discard-75%：丢前 75% 的 tool call 历史。
  • Discard-all：完全清空 tool 历史（类似 Anthropic 的 new-context tool）。
  • Parallel-fewest-step（baseline）：并行采 N 条独立 trajectory，选 step 数最少的。
Figure 6. 横轴 Real Steps（100–900），纵轴 BrowseComp 准确率（52.5–70）。蓝 Summary 曲线缓爬至约 60.2（≈ 364 步）；橙 Discard-75% 仅在 100–150 步小段、终于 57.5；绿 Discard-all 从 ~120 步起爬升迅速，到 ~400 步达 67.6；灰 Parallel-fewest-step 用更多步数（>900）才达到约 68.5。结论：Discard-all 在效率（step 数）和 scalability（最终分数）上同时占优——用大约一半步数就能逼近并行 baseline，而 Summary 虽然能涨分但效率最低。

5 深度分析：这篇论文真正的"信号"

  • 架构 × 训练 × 数据三件事都做到位才是开源能追的关键。 单做 sparse attention 不够（很多模型的 sparse attention 拿不到长上下文性能持平），单做更大 RL 也不够（容易塌方）。V3.2 把三件事打成一个 pipeline：DSA 让 RL rollout 便宜→ Scalable GRPO 让 RL compute 真能放大→ Synth pipeline 让 RL 能持续吃到难且可验证的数据。
  • "长对话里保留 thinking"是 thinking-in-tooluse 的工程关键。R1 风格的"每条新 message 就丢 reasoning"在 agent 场景下损耗巨大，V3.2 改写规则后才使 thinking-mode 在 tool-use 上具备性价比。
  • Hard to solve, easy to verify 是 RL 数据合成的核心节奏。General agent 通过"任务+solution+verifier 一起合成、solution 只能调 tool"把 verifier 锁死，从根本上避免了 reward hacking。
  • Test-time compute 扩展不只一种：序列化 (context mgmt) 与并行化 (parallel sampling) 各有 efficiency-scalability 权衡。Discard-all 说明"激进遗忘 + 重启探索"在 BrowseComp 上比 Summary 更有效。
  • Length penalty 的 trade-off 第一次被透明披露：V3.2 因为 length penalty 比 Speciale 弱很多分，token efficiency 仍是首要未解。

6 局限性与展望（论文自陈）

  • 世界知识广度仍弱于 Gemini-3.0-Pro，主因 pre-train 总 FLOPs 不足，未来计划继续扩 pre-train。
  • Token efficiency 弱于前沿闭源——同样质量需要更长链路，未来重点是"intelligence density"。
  • Hardest tasks（HLE、ICPC WF 极难题、MCP 长尾）仍落后，需要继续优化 base model 与后训练 recipe。
  • 当前 thinking 模式与 Roo Code/Terminus 不兼容，部署时需选 non-thinking 或换框架。
  • RL on long context 仍需 context-management 才能发挥（128K 在 search/MCP 不够用）。

7 核心总结

  1. DSA = lightning indexer + top-k token select，把核心 attention 从 $O(L^2)$ 砍到 $O(Lk=2048)$，从 V3.1-Terminus 通过 2.1B + 943.7B token 的 continued pre-train 平滑迁移。
  2. RL compute > 10% pre-train cost，靠 Unbiased KL / Off-Policy Mask / Keep Routing / Keep Sampling Mask 四件事压住 GRPO 不稳定。
  3. Thinking 在 tool-use 中保留仅在 user message 切换时丢弃 reasoning，使 thinking-mode 在 agent 场景具备实际收益。
  4. 1827 个合成环境 + verifier 把 RL 数据规模化：hard-to-solve, easy-to-verify，且能 transfer 到 $\tau^2$ / MCP 真实环境。
  5. Speciale 拿 IMO/IOI/CMO/ICPC WF 2025 金牌，但 token 效率代价大；正式 V3.2 用 length-penalty 换取部署经济性。
