引言 · 一个判断的陷阱

大多数人盯着一条 loss 曲线下降就安心了。但训练一个大模型，真正的工程问题不是"loss 在降吗"，而是三个更尖锐的问题：除了 loss 我还该看什么？loss 更低就一定更好吗？那些论文里的"质变"（比如 grokking）出现得早，是不是说明我训得更好？

本报告的核心立场：没有任何单一指标能定义"更好"。loss 会因 tokenizer、数据污染、捷径学习、分布失配而与真实能力解耦；grokking 的早晚是一个可调旋钮而非质量信号。真正的实践是一组指标交叉判读，并始终警惕"度量方式制造的假象"。

下面每一个反直觉的结论，都标注了它的成立条件和经过对抗式核验后的修正/打折——因为很多被广泛传播的论断其实被夸大了。

术语速查表

正文会反复用到这些概念，先把新定义讲清楚。

LM Loss（语言建模损失 / 交叉熵）
模型对下一个 token 预测的负对数似然 $\mathcal{L}=-\frac{1}{T}\sum_t \log p_\theta(x_t\mid x_{<t})$。困惑度 $\text{PPL}=e^{\mathcal{L}}$ 是它的指数形式。关键陷阱：它是 per-token 的，token 切得越碎、每个 token 越好预测，loss 数值越低——所以跨 tokenizer 不可直接比较。
Bits-per-Byte（BPB，每字节比特数）
把交叉熵除以文本的字节数而非 token 数：$\text{BPB}=\mathcal{L}_{\text{nats}}/(\ln 2 \cdot \text{bytes/token})$。这是一个tokenizer 无关的压缩率度量，让不同词表的模型可以公平比较。"压缩即智能"的近线性关系正是用 BPB 而非 raw loss 建立的。
Grokking（延迟泛化）
一种现象：训练精度早已达到 100%、模型严重过拟合很久之后，验证精度才突然从随机水平跃升到接近完美。两个必须区分的子概念：
· Onset timing（起始时刻）= 从"记住"到"泛化"中间的延迟有多长。本报告论证：这是一个可调旋钮。
· Occurrence（转变本身）= memorization→generalization 这个事件是否发生、发生后表示质量如何。这才是与能力相关的真实事件。
Anti-grokking（反 grokking）
把训练继续推到极端长（$10^6\sim10^7$ 步）+ 强 weight decay 后，已经 grok 的模型泛化又崩回随机。它的存在说明"训得越久越好"同样是错的——timing 与质量非单调。
Critical Batch Size（CBS，临界批量大小）
存在一个 batch 阈值：低于它，增大 batch 几乎线性减少所需步数（"完美并行"）；高于它，收益急剧递减。CBS 给出"加机器还划算"的上限。实证轨迹是近 0 → 快速上升 → plateau，且主要随数据量而非模型规模缩放。
Gradient Noise Scale（GNS，梯度噪声尺度）
McCandlish 2018 提出的廉价统计 $\mathcal{B}_{\text{simple}}=\text{tr}(\Sigma)/\|G\|^2$（梯度协方差迹除以梯度范数平方），用作 CBS 的代理。核验警告：在 Adam / 病态梯度下它会把 CBS 低估数个量级，不能单凭它精确定 batch。
z-loss（logit 正则）
辅助损失 $\mathcal{L}_z=\lambda\,(\log Z)^2$，其中 $Z=\sum_i e^{z_i}$ 是 softmax 的归一化分母。它把 $\log Z$ 拉向 0，防止 logit 量级膨胀导致数值发散。$\lambda$ 常取 $10^{-4}$。PaLM/ST-MoE/OLMo 用，GPT-3/LLaMA 系列不用。
MFU（Model FLOPs Utilization，模型算力利用率）
实际达到的 per-GPU FLOP/s 占理论峰值的比例。衡量"硬件用得有多满"，健康值约 50% 上下。持续偏低 = 并行/通信/数据加载有瓶颈，与模型质量无关但直接决定成本。
MaxVio（最大负载违背，MoE 专用）
专家混合（MoE）中，各专家实际负载相对均匀负载的最大偏离。过高意味着 routing collapse（少数专家被挤爆、其余闲置），浪费容量与 GPU。
Double Descent（双下降）
测试误差随模型规模/训练轮数先降→升→再降的非单调曲线，峰值出现在"插值阈值"（模型刚好能记住全部训练集）附近。核验：它主要发生在阈值附近或含标签噪声时，前沿 LLM 处于过参数化区，test loss 对 compute 近似单调，所以不是预训练的主导 regime。

Q1 · 除了 LM loss，该实时监控哪些指标？

分四类。总原则：没有单一指标是万能告警，多数信号是"有依据的先行指标"而非闭环自动干预器；并且很多指标存在"急性发散 vs 良性漂移"的语义混淆，必须结合上下文判读，否则会大量误报。

A. 稳定性信号（防发散）

指标测什么健康病态处理
梯度全局范数全模型梯度 L2 范数平稳/缓降，与 loss 同步与 loss spike 同时的突增 = 真发散跳过/回退 batch、降 LR
attention-logit / 线性层输出范数attention logits 与 QKV/Proj/FC2 输出 L2有界、缓增无界增长，高 LR 下最快qk-layernorm、softmax capping（预防性）
per-head attention entropy每个注意力头的分布熵适中熵塌缩伴随振荡/发散σReparam（谱范数重参数化）
z-loss / log(Z)softmax 归一化分母是否≈1$\log Z\approx0$logit 量级膨胀（高 LR）加 z-loss（$10^{-4}\cdot\log^2 Z$）
⚠ 对抗核验修正
  • 梯度范数末期缓升是良性的。 训练末期由 weight-decay × normalization × LR-schedule 交互导致的 grad-norm 缓慢爬升 [2506.02285] 是优化器配置次优的信号，loss 仍在收敛——当成发散告警就会误报。用 AdamC 类修正，而非中断训练。只有"与 loss spike 同时的突增"才是真发散。
  • attention entropy 与 attention-logit 范数不是独立告警。 由定理，熵随 attention-logit 谱范数指数下降 [2303.06296]，二者是"同一现象的两个视角"、共变。别把它们当两个独立证据。
  • z-loss 常见但非普适。 PaLM/ST-MoE/OLMo 采用，GPT-3/LLaMA 系列等许多生产模型并不使用——它是高 LR regime 的预防措施，不是必备项。

B. 批量 / 优化器信号

  • GNS（梯度噪声尺度）→ 估 critical batch size，决定数据并行/batch 上限。⚠ 核验：GNS 在 Adam/病态梯度下作为 CBS 代理不可靠（[2505.23971] 实测低估数个量级），可靠的 CBS 需要 branched runs，成本高。
  • gradient variance（梯度方差）→ 触发 sequence-length warmup。早期长序列样本驱动的极端梯度方差与不稳定强相关 [2108.06084]。注意补救手段是序列长度 warmup而非 batch warmup。

C. MoE 健康度（仅 MoE 模型）

  • per-expert load balance（MaxVio / 变异系数）：负载不均 → routing collapse、容量与 GPU 浪费。用动态 routing bias 修正（DeepSeek Loss-Free [2408.15664]）或 auxiliary load loss。配套监控 router z-loss 防 logit 爆炸。
  • ⚠ router entropy / specialization 多为事后诊断指标，不适合做每步告警。

D. 系统效率

  • MFU / 吞吐：持续低于预期 = 系统瓶颈（并行/通信/数据加载）。健康约 ~52% 峰值。术语由 PaLM 提出、[2211.05102] 形式化。它不反映模型质量，但直接决定烧钱速度。

Q2 · LM loss 更低，一定 = 模型更好吗？

核验后结论：不一定。loss 是下游能力的"强但非普适"代理。

判断的关键在于你是不是在做 apples-to-apples（同配方） 比较。

✅ 成立条件（同配方）

固定语料、tokenizer、架构时，loss 更低基本就是更好：

  • 相同 pretraining loss 的不同规模模型，下游表现一致，且能力在 loss 低于某阈值时才涌现 [2403.15796]。
  • 跨 31 个公开 base 模型，用 tokenizer 无关的 bits-per-byte 压缩率与平均基准分近乎线性相关 [2404.09937]——"压缩即智能"。

即：在同分布、未做对齐微调的 base 模型上，loss 更低确实更聪明。

❌ 解耦的情形（跨配方 / 有污染）

⚠ 对抗核验：几条流行"反例"被打了折扣（诚实呈现）
  • "39% 不成立"略夸大。 那 39% 衡量的是"是否存在可外推的线性形式"，而非"方向性失败"。真正反向（inverse/非单调）的占比小得多——大多数情形 loss 降仍对应能力升，只是非线性。
  • "同 loss 差 15%"有水分。 那个更优模型实际 OOD loss 低了 0.012（并非完全同 loss），且 15% 是单一基准的上界、由刻意工程化的优化器得到。
  • tokenizer 的 2.5× 差异是跨语言不是跨模型。 英语 ~1.2 vs 希腊/马耳他 ~3.1 tok/word；同语言跨模型差异小得多。且压缩≈智能仅对共享语料的 base 模型成立，instruction-tuning/RLHF 会升高压缩 loss 却改善对齐。
  • 记忆驱动的 loss 下降常是真实学习。 对长尾自然分布，记忆稀有样本是近优泛化所必需；只有高重复字符串的逐字背诵才是少量、非主导成分。不能一律当作"假捷径"。
  • 校准/多样性退化应归咎 post-training。 进一步压熵牺牲校准与多样性，但这主要由 RLHF/对齐造成——base 模型 loss 更低通常校准更好。这个反例不该算在"预训练 loss"头上。
  • 双下降不是预训练主导 regime。 它发生在插值阈值附近/含标签噪声，前沿 LLM 在过参数化区，scaling law 使 test loss 对 compute 近似单调。
给从业者的可操作判断：当你看到 loss 降了，先问三件事——(1) tokenizer/语料变了吗？变了就用 BPB 比，别比 raw loss；(2) 评测集可能进训练数据了吗？查污染；(3) 这是 base 模型还是对齐后的？对齐会让 loss 与"好用"反向。三关都过，loss 更低才能放心说更好。

Q3 · grokking 更早出现，代表模型/训练更好吗？

核验后结论：不能。grokking 的"延迟时长"主要是一个可被原则性放大或彻底消除的可调旋钮，而非模型质量的因果指标。

你提到 grokking 常出现在"做选择题之类"的任务上——这里要先提醒：选择题准确率正是一种离散指标，它本身就容易把平滑的概率改善压成阶跃假象。用它判断 grokking 要尤其小心（见 Q2 的度量陷阱）。

必须区分的两件事

Timing
延迟多长 = 旋钮，可工程化掉
Occurrence
转变本身 = 真实表示质量事件
  1. 延迟时长（timing）是旋钮。 多篇理论证明它可被超参/初始化/数值技巧原则性地放大或消除：岭回归闭式证明 [2601.19791]、Omnigrok 的 LU 机制 [2210.01117]、edge-of-numerical-stability [2501.04697]。Grokfast 通过放大慢梯度加速 >50× [2405.20233]。既然能随手调，"更早 grok"本身就不携带因果性的"更好"。
  2. memorization→generalization 转变是真实事件。 转变后概念向量更具预测性、占据更高维子空间，伴随真实电路/结构形成。延迟可调，但转变本身与能力相关。
⚠ 关键：timing 与质量非单调

把旋钮推得更远会触发 anti-grokking——超长训练后泛化崩回随机。所以既非"越早越好"，也非"越久越好"。

还要警惕诊断指标本身的可靠性：权重 L2 范数在 canonical 模算术设定下是机制性的，但在真实/多属性设定下不可靠。激活稀疏性、特征秩、commutator defect 等相关性更强——但其中只有 commutator defect 有干预性因果证据，sparsity/circuit-complexity 在 anti-grokking regime 会失效 [2405.12755]。

哪些旋钮能控制 grokking 起始？

  • weight decay 与 learning rate：Norm-Separation Delay Law，$R^2\approx0.9$。
  • 优化器选择：Muon 相比 AdamW 把 onset 从 ~153 → 103 epoch 提前 [2504.16041]。
  • 初始化尺度 / 权重范数：Omnigrok 既能诱导也能消除 grokking [2210.01117]。
  • 归一化与架构拓扑：>20× 缩短延迟，且无需 weight decay。
  • 数值稳定化：StableMax/⊥Grad 在无 weight decay 下也能快速泛化 [2501.04697]。

本质机制：过参数化 + 小数据下，记忆电路与泛化电路竞争，存在 critical dataset size 阈值，跨过阈值越早、onset 越早——而这恰恰可被工程化掉。

它在真实 LLM 预训练里重要吗？

grokking 确实出现在真实大规模预训练中，但形态不同：是局部、异步的——不同数据子群按各自分布在不同时间进入 memorization→generalization 转变（OLMoE 单 epoch MoE 预训练 [2506.21551]；dense LM 语法泛化延迟 [2606.00230]）。

⚠ 必须强 hedge 的三点
  • 证据薄弱：MoE 那篇是 n=1 单模型研究，标题自带问号，pathway 解释为相关性而非因果定律。
  • 归因错位：局部/异步源于异构单遍数据而非 MoE 架构本身（dense LM 也有），不应归因于 MoE。
  • regime 不匹配：经典 grokking 在单 epoch web-scale 预训练中几乎不显现（要人为构造 train/val 代理划分才观测到）；anti-grokking 所需的极端过训练（$10^6\sim10^7$ 步 + 强 weight decay + 微小数据）正好与 compute-optimal/欠训练的标准预训练相反，实践中很少触及。
底线：grokking 不是纯玩具任务伪影，但"更早 grok = 预训练更好"在真实 LLM 中并无证据支撑——因为这些工作恰恰显示的是泛化滞后于记忆。把 grokking 起始时间当 KPI 去优化，是优化了一个旋钮，不是优化了模型。

Q4 · 训练中可"即时查看"的诊断 / 预测方法

必须诚实区分两类，因为很多被称作"早期诊断"的方法其实是训练前规划工具而非在线监控：

  • (a) 真正的在训实时统计：梯度/logit 范数、噪声尺度、loss 曲线机制信号——每步几乎零成本可算。
  • (b) 跨规模外推预测：在小模型族上拟合，训练前/早期对大模型做 go/no-go——精度高度依赖任务与指标。

总结 · 一页实操清单

① 仪表盘该放什么（按优先级）

  • 主信号：train/val loss + per-domain 验证 loss（防某个数据域退化）
  • 稳定性：梯度全局范数、max attention-logit、（高 LR 时）log(Z)
  • 批量：GNS 轨迹（当趋势看，不当精确值）
  • MoE：MaxVio + router z-loss
  • 系统：MFU / 吞吐

② "loss 降了"先过三关：tokenizer/语料变没变（变→用 BPB）、有没有数据污染、是不是对齐后模型。

③ 别把 grokking 起始时间当 KPI：它是旋钮不是质量。要看就看转变是否发生及转变后的表示质量，而非多早发生。

④ 区分在线监控 vs 训练前规划：梯度/logit 范数是前者；FLP/PassUntil/scaling law 外推是后者。

⑤ 警惕离散指标的假象：选择题准确率、exact-match 会把平滑改善压成"突变/涌现"。判断质变前，先换连续指标（BPB、编辑距离、per-token log-prob）复核。
